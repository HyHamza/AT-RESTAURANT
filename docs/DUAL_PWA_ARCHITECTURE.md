## Key Architectural Decisions

### 1. Unique Manifest IDs
- **Why**: Browser uses ID to differentiate apps
- **Impact**: Enables independent installation
- **Critical**: Must be different for each app

### 2. Separate Scopes
- **Why**: Prevents service worker conflicts
- **Impact**: Each SW only handles its routes
- **Critical**: User `/`, Admin `/admin`

### 3. Route-Based Filtering
- **Why**: Ensures SWs don't interfere
- **Impact**: Clean separation of concerns
- **Critical**: User SW ignores `/admin`, Admin SW only handles `/admin`

### 4. Dynamic Manifest Switching
- **Why**: Admin pages need different manifest
- **Impact**: Correct manifest loaded per context
- **Critical**: AdminHead component updates manifest link

### 5. Separate Icon Sets
- **Why**: Visual differentiation on homescreen
- **Impact**: Users can easily identify each app
- **Critical**: Different icons for user vs admin

## Summary

The dual PWA architecture achieves complete independence through:

1. âœ… **Unique manifest IDs** - Browser recognizes as different apps
2. âœ… **Separate scopes** - No service worker conflicts
3. âœ… **Route filtering** - Each SW handles only its routes
4. âœ… **Dynamic manifest** - Correct manifest per context
5. âœ… **Separate caches** - No cache conflicts
6. âœ… **Different icons** - Visual differentiation

This enables both apps to be installed on the same device and run independently without any conflicts.
